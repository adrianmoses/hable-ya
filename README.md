
### hable-ya

A single, on-device voice model that is simultaneously the conversational partner, the assessor, and the adaptive engine


Based on the ideas of comprende-ya and habla.practice this is a voice-only agent for improving Spanish


This project will also include a process for supporting other source and target languages.



#### Core Components

-- Gemma 4 LLM fine-tuned for adaptive comprehension and teaching 
-- Pipecat pipeline for STT -> LLM -> TTS (llama.cpp does not support audio encoders)
-- FastAPI and websockets for hands-off conversations
-- llama.cpp for model serving
-- knowledge graph based learner model


A key idea is a knowledge graph based learner model. A dynamic graph to identify strengths, weakness, the current level and how the learner is progressing.
The learner model will be established and modified by the agent via tools. 


#### Roadmap

1. Build agent loop with Gemma 4 E4B, pipecat and llama.cpp

#### Model Eval

The initial eval is to determine if the model needs to few-shot learning or fine-tuning

1. pedagogical score

This is an average score of a few heuristics

- `recast_present` agent is not repeating errors and is instead responding with correction
- `recast_explicit` agent explicity corrects the learner (negative weight)
- `register_correct` agent responded at a level or slightly above than the learner
- `sentence_count_ok` agent responded with a balance of conciseness and expressiveness
- `question_count_ok` agent responded with a question
- `L1_in_response` agent responsed in English
- `error_repeated` agent repeats errors from the learner


2. tool fidelity score

- `log_turn_called` agent calls tool on each conversation
- `tool_args_correct` agent calls with correct args


3. Cold start score
- `band_accuracy` agent adapts to user level

4. Composite
- (`pedagogical` x 0.7) + (`tool_fidelity` x 0.3)


A `compare.py` will be used to compare differences across base, prompted, and whether to use fine-tuned




#### Agent Eval

The agent eval will be done by a more advanced LLM 

- Synthetic learner simulation (with error patterns and probability to commit them)
- Session outcome judgement
  - pedagogical flow
  - level consistency
  - recast naturalness
  - learner production space (was the learner given enough space to produce)
  - session coherence (did the session feel like natural conversation or disjointed exercises)




Test fixtures will also be generated by Opus but will need human review

#### Fine-tuning for pedagogical conversation


Model Eval decision framework


Overall baseline score:

Score > 0.80   → good enough for v1, move on
Score 0.65-0.80 → try prompt engineering first
Score < 0.65   → prompt engineering unlikely to close the gap, fine-tune

Dimension thresholds:

recast_present < 0.70       → fine-tune
                              This is core behavior. If the model isn't
                              recasting 70% of the time after prompt
                              engineering, prompting alone won't fix it.
                              The model needs examples of what good looks like.

recast_explicit > 0.20      → fine-tune
                              If 1 in 5 responses is explicitly correcting
                              the learner after prompt iteration, the model
                              has a strong prior toward correction that
                              prompting can't fully override.

register_correct < 0.70     → fine-tune
                              Especially if A1/A2 scores are low — base
                              models tend to default to a natural conversational
                              register that's too complex for beginners.

L1_in_response > 0.15       → fine-tune
                              If the model switches to English more than
                              15% of the time when the learner uses English,
                              this is a strong prior that needs examples
                              to override.

sentence_count_ok < 0.75    → prompt engineering first
                              Verbosity usually responds to instruction.
                              Only fine-tune if it persists after prompting.

question_count_ok < 0.80    → prompt engineering first
                              Same — this is a behavioral constraint that
                              explicit instruction handles well.

error_repeated > 0.05       → fine-tune immediately
                              Any rate above 5% of turns where the agent
                              reproduces a learner error is a hard failure.
                              This actively harms acquisition.


Per-band breakdown:

```

# A model that scores 0.75 overall might look like this:

by_band = {
    "A1": 0.52,    # ← fine-tune needed — mimicry and register hardest
    "A2": 0.68,    # ← borderline, prompt engineering first
    "B1": 0.79,    # ← acceptable
    "B2": 0.81,    # ← good
    "C1": 0.83,    # ← good — base model comfortable at native register
}

# The overall 0.75 looks ok but A1 is genuinely broken.
# Fine-tuning dataset needs heavy A1/A2 weighting.
```



Cold start thresholds:

```
band_accuracy < 0.75        → fine-tune cold start examples
MAE > 0.20                  → fine-tune cold start examples

These thresholds matter because a bad cold start poisons
the entire learner profile — every subsequent session
runs at the wrong level.
```


#### Fine-tuning datasets

Three dataset types:
1. SFT examples          → teach correct behavior from scratch
2. DPO contrast pairs    → teach what to avoid (uses negative examples
                           from fixtures directly)
3. Tool call examples    → teach reliable function calling


SFT is a two-turn conversation example

```json
{
  "messages": [
    {
      "role": "system",
      "content": "<rendered system prompt for this level>"
    },
    # Prior context — 1-3 turns minimum
    # More context = better, but watch token budget
    {
      "role": "assistant",
      "content": "¿Qué hiciste el fin de semana pasado?"
    },
    {
      "role": "user",
      "content": "Fui al mercado y compré mucho verdura fresca."
    },
    # Target completion — what we want the model to learn
    {
      "role": "assistant",
      "content": "¡Qué bien! ¿Encontraste muchas verduras de temporada? \n\n[TOOL_CALL: log_turn]{\"learner_id\": \"...\", \"learner_utterance\": \"Fui al mercado y compré mucho verdura fresca.\", \"L1_used\": false, \"errors_observed\": [{\"type\": \"grammar\", \"example\": \"mucho verdura\", \"recast\": \"muchas verduras\"}], \"vocab_produced\": [\"mercado\", \"verdura\", \"fresca\"], \"fluency_signal\": \"moderate\"}"
    }
  ],
  "metadata": {
    "category":   "single_error_recast",
    "cefr_band":  "A2",
    "error_type": "gender_agreement",
    "difficulty": "straightforward",
    "weight":     1.0    # upweight rare or critical examples
  }
}
```

these can be used for tool call examples as well

DPO pairs can come from negative fixtures

```json
{
  "messages": [
    {"role": "system",    "content": "<system prompt>"},
    {"role": "assistant", "content": "¿Qué hiciste el fin de semana?"},
    {"role": "user",      "content": "Fui al mercado y compré mucho verdura fresca."}
  ],

  # What the model should produce
  "chosen": {
    "role": "assistant",
    "content": "¡Qué bien! ¿Encontraste muchas verduras de temporada?\n\n[TOOL_CALL: log_turn]{...}"
  },

  # What the model should not produce
  "rejected": {
    "role": "assistant",
    "content": "Dijiste 'mucho verdura' pero es 'muchas verduras'. ¿Fuiste solo?"
  },

  "metadata": {
    "rejection_reason": "explicit_correction",
    "cefr_band": "A2"
  }
}
```

Tool calls can be in a focused set

```json
{
  "messages": [
    {"role": "system",    "content": "<system prompt>"},
    {"role": "assistant", "content": "¿Cómo fue tu semana?"},
    {
      "role": "user",
      "content": "Estuvo bien, trabajé mucho pero también salí con amigos."
      # No errors — clean utterance, just teaching log_turn structure
    },
    {
      "role": "assistant",
      "content": "¡Qué bien equilibrado! ¿Adónde fuiste con tus amigos?\n\n[TOOL_CALL: log_turn]{\"learner_id\": \"...\", \"learner_utterance\": \"Estuvo bien, trabajé mucho pero también salí con amigos.\", \"L1_used\": false, \"errors_observed\": [], \"vocab_produced\": [\"trabajé\", \"salí\", \"amigos\"], \"fluency_signal\": \"strong\"}"
    }
  ],
  "metadata": {
    "category":  "tool_call_correctness",
    "cefr_band": "B1",
    "has_error": false
  }
}
```


Use band and category distribution to influence the training examples. Target 3k-4k training examples


Exmaple order of example generation

```
Step 1 → 200 examples across all categories and bands
          Run a quick fine-tune on E4B (1-2 epochs)
          Run model eval — compare to baseline

Step 2 → Identify which dimensions moved and which didn't
          Generate 800 more examples weighted toward
          dimensions that didn't move

Step 3 → Full fine-tune (3-4 epochs)
          Run full model eval + cold start eval

Step 4 → If DPO needed, generate contrast pairs
          from fixture negatives + worst SFT failures
          Run DPO fine-tune on top of SFT checkpoint
```
